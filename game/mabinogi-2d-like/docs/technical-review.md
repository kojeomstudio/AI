# Mini-MMO 기술 검토 (2D 마비노기 스타일)

> 작성일: 2026-06-13 · 상태: 검토(Proposal) · 경로: `game/mini-mmo/`

## 0. 결론 먼저

**가능합니다.** 단, "미니멀"이라도 본질은 MMO(실시간 다중 접속 + 영속 데이터)이며,
이는 게임 장르 중 난이도가 높은 축에 속합니다. 핵심은 **범위를 좁히고
수직 슬라이스(vertical slice)부터 완성**하는 것입니다.

권장 스택은 레포의 기존 C# 자산을 그대로 활용하는 **올(all)-C# 구성**입니다.
서버·클라이언트·공유 프로토콜을 한 언어로 묶으면 1인/소규모 개발에서
가장 큰 비용 절감 효과가 납니다.

## 1. 권장 기술 스택

| 영역 | 권장 | 이유 / 대안 |
|------|------|-------------|
| 클라이언트 | **Godot 4 (C#/.NET)** | 2D 우선 설계, 가벼움, 빠른 이터레이션, 무료. C# 스킬 재사용. 대안: 웹 배포 원하면 TS+Phaser, 고사양이면 Unity |
| 서버 | **.NET (ASP.NET Core)** — Auth/REST + 실시간 게임 서버 | 권위(authoritative) 서버. 기존 .NET 8 자산과 동일. 대안: Node+Colyseus(룸 기반 MMO 특화) |
| 실시간 전송 | **WebSocket + MessagePack(바이너리)** | .NET 내장, 데스크톱·웹 모두 호환, 디버깅 쉬움. 업그레이드 경로: LiteNetLib(UDP, 저지연) |
| 영속 DB | **PostgreSQL** (계정·캐릭터·인벤토리) | 트랜잭션·관계형. 개발 초기엔 **SQLite**로 시작 가능 |
| 캐시/세션 | **Redis** | 세션 토큰, 접속 상태, 존(zone) 휘발 상태, 랭킹 |
| ORM | **EF Core** | C# 마이그레이션·스키마 관리 |
| 공유 코드 | **C# class library** (프로토콜/DTO/상수) | 서버·클라가 같은 타입 공유 → 직렬화 불일치 버그 제거 |

> 핵심 판단: 마비노기 자체가 TCP 기반이고 미니멀 규모(수~수십 명)에서는
> WebSocket/TCP로 충분합니다. UDP는 빠른 액션 전투를 큰 규모로 돌릴 때만 필요.

## 2. 전체 아키텍처

```
┌────────────┐   HTTPS/REST   ┌──────────────┐
│  Client    │───────────────▶│  Auth/API    │  로그인, 캐릭터 생성/선택
│ (Godot C#) │   (로그인,      │ (ASP.NET)    │  → JWT 발급
│            │    캐릭터목록)   └──────┬───────┘
│            │                        │
│            │   WebSocket(MsgPack)   ▼
│            │◀──────────────▶┌──────────────┐   ┌────────────┐
└────────────┘   실시간 상태   │ Game Server  │──▶│ PostgreSQL │ 영속 저장
                 동기화        │ (틱 루프)     │   └────────────┘
                              │ 권위 시뮬레이션│   ┌────────────┐
                              └──────┬───────┘──▶│   Redis    │ 세션/존 상태
                                     │            └────────────┘
                              ┌──────▼───────┐
                              │ Shared.Proto │  서버·클라 공유 DTO/상수
                              └──────────────┘
```

**권위 서버(authoritative)가 핵심 원칙**: 모든 게임 로직(이동 검증, 전투 판정,
아이템 변동)은 서버가 진실의 원천. 클라이언트는 입력 전송 + 렌더 + 예측만 담당.
영속 데이터와 다중 접속이 있는 순간 클라이언트를 신뢰하면 안 됩니다.

## 3. 서버 설계

- **2개 프로세스로 분리**:
  - *Auth/API 서버*: 무상태 REST. 로그인 → 패스워드 검증 → JWT 발급, 캐릭터 CRUD.
  - *게임 서버*: 상태 보유(stateful). WebSocket 연결 수락, JWT 검증, 틱 루프 구동.
- **틱(tick) 루프**: 미니멀 기준 **15~20Hz**(50~66ms). 매 틱마다
  입력 처리 → 시뮬레이션 갱신 → 관심영역(AOI) 내 플레이어에게 스냅샷/델타 브로드캐스트.
- **존(zone)/룸 모델**: 맵 1개 = 1 존. 존마다 독립 상태와 플레이어 목록.
  미니멀은 단일 존으로 시작, 이후 채널/인스턴스로 확장.
- **메시지 종류**: `Join/Leave`, `Move`(입력 의도), `EntitySnapshot`(위치/HP),
  `CombatEvent`, `ChatMessage`, `Ping`. 모두 Shared 라이브러리의 타입으로 정의.

## 4. 클라이언트 설계 (Godot 4 / C#)

- **레이어 분리**: 네트워크 클라이언트(WebSocket) · 게임 상태(서버 스냅샷 캐시) ·
  렌더(Node2D 씬) · 입력.
- **네트워크 보정**:
  - *클라이언트 예측*: 내 캐릭터는 입력 즉시 이동 후 서버 확정값과 대조·보정.
  - *원격 보간(interpolation)*: 다른 플레이어는 최근 두 스냅샷 사이를 보간해 부드럽게.
  - 미니멀 초기엔 예측 없이 "서버 확정 위치만 보간"으로 시작해도 충분(지연 체감↑이나 단순).
- **타일맵 기반 2D**: Godot TileMap + 스프라이트. 마비노기풍은 쿼터뷰지만,
  미니멀은 탑다운 2D로 시작하고 비주얼은 나중에.

## 5. DB 설계

- **PostgreSQL (영속)** — 트랜잭션이 중요한 데이터:
  `accounts`(id, email, password_hash), `characters`(id, account_id, name, level, exp, pos_x, pos_y, map_id, hp),
  `inventory_items`(id, character_id, item_def_id, qty, slot). 정적 데이터(아이템/몬스터 정의)는
  JSON/테이블로 관리(레포의 ExcelToJsonExporter 패턴 재사용 가능).
- **Redis (휘발/세션)**: 로그인 세션 토큰, 온라인 플레이어 맵, 존별 임시 상태, 채팅 레이트리밋.
- **저장 정책**: 게임 서버는 메모리에서 시뮬레이션하고 **주기적(예: 30초)·이벤트성(로그아웃, 중요 거래)**
  으로 Postgres에 영속화. 매 틱 DB 쓰기는 금지(병목).
- **개발 초기**: Postgres 대신 **SQLite**로 시작 → EF Core 마이그레이션은 동일하게 작성되어 전환 쉬움.

## 6. 권장 범위 (미니멀 = 수직 슬라이스)

**먼저 만들 것 (MVP):**
1. 로그인/회원가입 → JWT
2. 캐릭터 생성·선택
3. 단일 공유 맵 입장
4. 다른 플레이어 실시간 이동 동기화 ← *멀티플레이의 심장, 가장 먼저 검증*
5. 몬스터 1종 + 기본 전투(공격 → HP 감소 → 처치 → 경험치)
6. 캐릭터 상태 저장/복원

**나중으로 미룰 것:** 채널/인스턴스, 생활 스킬, 거래, 길드, 파티, 안티치트,
샤딩/스케일링, 정교한 전투 콤보 시스템.

## 7. 디렉터리 구조 (제안)

```
game/mini-mmo/
├── docs/                  # 본 문서, 프로토콜 명세, 결정 기록
├── server/
│   ├── MiniMmo.Auth/      # ASP.NET Core REST (로그인/캐릭터)
│   ├── MiniMmo.Game/      # 실시간 게임 서버 (틱 루프, WebSocket)
│   └── MiniMmo.Data/      # EF Core DbContext, 마이그레이션
├── shared/
│   └── MiniMmo.Shared/    # 프로토콜 DTO, 상수, MessagePack 계약
├── client/                # Godot 4 프로젝트 (C#)
├── tools/                 # 정적 데이터(아이템/몬스터) 변환 등
└── docker-compose.yml     # Postgres + Redis 로컬 개발 환경
```

빌드 산출물은 레포 규칙대로 `Bins/mini-mmo-*/`로 출력.

## 8. 솔직한 리스크 / 주의

- **MMO는 본질적으로 어렵다**: 멀티플레이 동기화가 전체 일정의 대부분. 4번(이동 동기화)이
  안 되면 나머지는 의미 없으니 **여기부터 프로토타입**.
- **C# 단일 언어의 트레이드오프**: 웹 즉시 배포가 중요하면 Godot 웹 익스포트 또는
  TS 스택을 재고. 데스크톱 중심이면 C#이 최적.
- **확장성은 지금 설계하지 말 것**: 단일 존·단일 게임서버 프로세스로 시작.
  존 분리/샤딩은 동작하는 게임이 나온 뒤 리팩터링.
- **Godot의 C#**: GDScript보다 생태계 예제가 적음. 다만 .NET 자산 재사용 이점이 이를 상쇄.
